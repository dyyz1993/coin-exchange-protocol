# Issue 管理指南

## 📋 概述

本文档定义了金币交易协议项目的 Issue 管理流程，旨在提高团队效率、减少重复工作、保持项目清晰可见。

## 🎯 管理目标

1. **减少噪音**：关闭重复 Issues，让团队聚焦真实问题
2. **提升效率**：避免多人处理同一问题
3. **改善可见性**：准确反映项目真实问题数量
4. **规范流程**：建立清晰的 Issue 创建和维护流程

## 📊 Issue 分类体系

### 优先级标签

| 标签 | 含义 | 响应时间 | 示例 |
|------|------|---------|------|
| `P0` | 紧急/阻塞 | 立即处理 | 生产环境崩溃、安全漏洞 |
| `P1` | 高优先级 | 24小时内 | 核心功能缺陷、性能问题 |
| `P2` | 中等优先级 | 1周内 | 功能改进、优化建议 |
| `P3` | 低优先级 | 按计划处理 | 文档完善、小改进 |

### 类型标签

| 标签 | 含义 | 说明 |
|------|------|------|
| `bug` | 缺陷 | 功能不符合预期 |
| `feature` | 新功能 | 新增功能需求 |
| `improvement` | 改进 | 现有功能优化 |
| `documentation` | 文档 | 文档相关 |
| `maintenance` | 维护 | 技术债务、重构 |
| `question` | 疑问 | 需要澄清的问题 |

### 状态标签

| 标签 | 含义 | 说明 |
|------|------|------|
| `needs-triage` | 待分类 | 新创建的 Issue |
| `in-progress` | 进行中 | 已分配正在处理 |
| `blocked` | 阻塞中 | 被其他问题阻塞 |
| `needs-review` | 待审查 | 已完成待审查 |
| `wontfix` | 不予修复 | 明确不处理的问题 |
| `duplicate` | 重复 | 已存在的重复问题 |

## 🔄 Issue 生命周期

```
1. 创建 Issue → needs-triage
2. 分类和评估 → 添加优先级和类型标签
3. 分配责任人 → in-progress
4. 开发和修复 → 创建 PR
5. 代码审查 → needs-review
6. 合并和关闭 → 关闭 Issue
```

## 📝 Issue 创建规范

### 创建前检查

1. **搜索已存在的 Issues**
   - 使用关键词搜索
   - 检查类似问题描述
   - 确认不是重复 Issue

2. **选择正确的 Issue 模板**
   - Bug 报告：使用 `bug_report.md`
   - 功能请求：使用 `feature_request.md`

3. **填写必要信息**
   - 清晰的标题
   - 详细的问题描述
   - 复现步骤（如果是 Bug）
   - 预期行为 vs 实际行为
   - 环境信息

### Issue 标题格式

```
[标签] 简短描述

示例：
🔧 [P1] 修复空投超额领取问题
✨ [P2] 新增用户交易历史导出功能
📝 [P3] 完善 API 文档
```

## 🧹 定期清理流程

### 每周清理（周五下午）

1. **检查新创建的 Issues**
   - 标记重复 Issues
   - 关闭已解决的问题
   - 更新过时的 Issues

2. **更新 Issue 状态**
   - 标记停滞的 Issues
   - 更新优先级
   - 关闭不活跃的 Issues（30天无更新）

### 每两周清理（第一个和第三个周五）

1. **全面审查开放 Issues**
   - 识别新的重复 Issues
   - 合并相似 Issues
   - 重新评估优先级

2. **生成报告**
   - Issue 数量统计
   - 按优先级分布
   - 按类型分布
   - 平均解决时间

## 🎓 最佳实践

### 对于开发团队

1. **创建 Issue 前**：搜索至少 3 次不同关键词
2. **分配 Issue 前**：检查责任人当前任务数量（建议最多 2 个）
3. **关闭 Issue 前**：确保相关 PR 已合并
4. **更新 Issue 时**：添加进度评论，保持透明

### 对于项目管理者

1. **每周审查**：确保没有停滞的 Issues
2. **每月统计**：分析 Issue 趋势和解决效率
3. **季度回顾**：优化流程，更新标签系统
4. **持续改进**：收集团队反馈，改进流程
